## 1. Authentication & Sessions

### Purpose
Controls who can access the application. Uses DB-backed sessions (30-day expiry), bcrypt password hashing, and cookie-based session tokens. Auth is enforced via `requireAuth()` in every server action and protected layout.

### Main Entities
`User` (passwordHash, isActive), `Session` (expiresAt)

### Business Rules
- Login requires email + password
- Password compared with bcrypt hash stored in User
- Inactive users (`isActive = false`) are rejected even with valid credentials
- Users without a `passwordHash` are rejected (student-only accounts)
- Sessions expire after 30 days
- Expired sessions are deleted on access attempt
- All `(app)` routes are protected via layout-level `proxyAuth()`
- No role check beyond authentication: any authenticated user (TEACHER, STUDENT, ADMIN) can access all teacher functions

### Critical Validations
- `!user.passwordHash` → rejected (users created without password cannot log in)
- `!user.isActive` → rejected
- Session cookie presence + DB lookup + expiry check on every authenticated request

### Dependencies
Used by every module. Auth failure redirects to `/login`.

### Risks if Logic Breaks
- Authentication bypass → data exposure of all students, payments, charges
- Session not invalidated → former users retain access
- Role not checked → a student logging in can create/edit teacher data

## 2. Students

### Purpose
Teachers create and manage students. Students are scoped per teacher via `createdByTeacherId`.

### Main Entities
`User` (role=STUDENT, createdByTeacherId, email?, phone?)

### Business Rules
- Student requires firstName + lastName
- At least one contact method: email OR phone
- Email must be globally unique (across all users, all teachers)
- Student is scoped to the creating teacher (`createdByTeacherId = teacher.id`)
- Teacher can only view/edit their own students
- No student deletion action is implemented
- Students can exist without an email (phone only)

### Critical Validations
- `!firstName` → error
- `!lastName` → error
- `!email && !phone` → error
- `email` format validated by regex
- Email uniqueness checked before insert (not just DB constraint)

### Dependencies
- Charges: `userId` references student
- Payments: `userId` references student
- Packages: `userId` references student
- Lessons: `participants[].userId` references student

### Risks if Logic Breaks
- Teacher sees another teacher's students
- Email collisions across multi-teacher future scenario
- Student orphaned records when teacher is removed (no cascade guard)

## 3. Lessons & Calendar

### Purpose
Teachers schedule lessons with optional student participants. Calendar view shows lessons by day. Scheduling conflict detection prevents double-booking at the exact same timestamp.

### Main Entities
`Lesson`, `LessonParticipant`

### Business Rules
- Lesson requires title, scheduledAt (Zurich datetime), durationMin (positive integer)
- priceAmount is optional
- Teacher cannot schedule two lessons at the exact same timestamp (exact match only, no overlap)
- Conflict check on update excludes the lesson being edited (self-exclusion)
- Student participant is optional at lesson creation time
- Adding a participant: student must exist as role=STUDENT — ownership (createdByTeacherId) is NOT checked
- Duplicate participant rejected by DB unique constraint (P2002 handled as DomainError)
- Deleting a lesson cascades to LessonParticipants → PackageUsages — package minutes NOT restored (critical bug)
- Removing a participant cascades to PackageUsage — package minutes NOT restored (critical bug)
- Lesson deletion sets charge.lessonId to NULL (SetNull cascade on Charge)

### Critical Validations
- durationMin: positive integer, not float, not zero
- scheduledAt: must match `YYYY-MM-DDTHH:mm` format (calendar validity NOT checked)
- priceAmount: decimal regex if provided
- Conflict: exact timestamp match only (no duration-based overlap detection)

### Dependencies
- Calendar: lessons displayed by date
- Packages: PackageUsage links LessonParticipant → Package
- Charges: charge.lessonId can reference a lesson

### Risks if Logic Breaks
- Lesson deleted with active package assignments → package minutes silently lost
- Schedule conflict misses overlapping (not same-start) lessons
- Any student from any teacher can be added to a lesson

## 4. Packages

### Purpose
Teachers sell hour bundles (packages) to students. Each package creation automatically generates a linked Charge. Package minutes are consumed when assigned to lesson participants. Status transitions reflect consumption and expiry.

### Main Entities
`Package`, `PackageUsage`, `Charge` (auto-created on package creation)

### Business Rules
- totalHours must be a positive integer (converted to minutes: × 60)
- remainingMinutes initialized to totalMinutes on creation
- Expiry date must be in the future at creation and at update
- Package creation is atomic: Package + Charge created in one transaction
- Update adjusts remainingMinutes proportionally: `newRemaining = max(remaining + diff, 0)`
- Status: ACTIVE (has remaining minutes), EXHAUSTED (0 remaining), EXPIRED, CANCELED
- EXHAUSTED → ACTIVE when total hours increased and newRemaining > 0
- EXPIRED and CANCELED statuses preserved through updates (only EXHAUSTED reacts to updates)
- No deletion action

### Critical Validations
- totalHours: positive integer only
- amount: decimal with up to 2 decimal places
- expiresAt: must be in future if provided

### Dependencies
- Charges: Package creation auto-creates a Charge (type=PACKAGE)
- PackageUsage: Lessons consume package minutes
- Students: package belongs to one student

### Risks if Logic Breaks
- Atomic transaction failure: Package without Charge or vice versa
- Update total hours below consumed hours → inconsistent data
- Status stuck in EXHAUSTED after hours increase

## 5. Package Assignment (Lesson → Package)

### Purpose
Assigns a student's package to their participation in a lesson, consuming minutes from the package. Assignment and removal are atomic transactions with concurrency protection.

### Main Entities
`PackageUsage`, `Package`, `LessonParticipant`

### Business Rules
- Package must be ACTIVE status
- Package must not be expired (`expiresAt < now`, even if status=ACTIVE)
- Package must belong to the same student as the participant
- Package must have remaining minutes > 0
- Minutes consumed = `min(lesson.durationMin, pkg.remainingMinutes)`
- Atomic conditional update: `WHERE remainingMinutes >= minutesConsumed` prevents double-spend
- Only one package per participant (unique constraint on `lessonParticipantId`)
- Remove assignment: restores minutes with `Math.min(remaining + consumed, totalMinutes)` cap
- Remove: EXHAUSTED → ACTIVE after minute restoration; EXPIRED/CANCELED preserved

### Critical Validations
- `pkg.status !== ACTIVE` → DomainError "Package is not active"

## 6. Charges

### Purpose
Teachers issue charges to students for lessons, packages, adjustments, or other items. Payment allocations are linked to charges. Charge status is automatically derived from payment allocations.

### Main Entities
`Charge`, `PaymentAllocation`

### Business Rules
- Charge requires student (owned by teacher), title, amount, currency
- lessonId optional — must be teacher's lesson if provided
- amount must be a valid decimal (≤ 2 decimal places) — zero is technically accepted
- Status automatically derived from allocations on update (unless CANCELED is submitted)
- CANCELED is the only status a teacher can manually set that persists through update
- PENDING: totalPaid = 0 or no allocations
- PARTIALLY_PAID: 0 < totalPaid < amount
- PAID: totalPaid >= amount (over-payment is accepted)
- Amount reduction after payment can flip status to PAID
- No deletion action — charges are permanent records
- Currency defaults to CHF

### Critical Validations
- userId: STUDENT role, owned by teacher
- lessonId: belongs to teacher if provided
- amount: decimal format (but zero passes)
- Status recalculation overrides form-submitted status (except CANCELED)

### Dependencies
- Payments: PaymentAllocation links payments to charges
- Packages: Package creation auto-creates a PACKAGE-type charge
- Students: userId references student
- Lessons: lessonId optionally references a lesson

### Risks if Logic Breaks
- Charge status inconsistent with actual payments
- Over-payment silently accepted
- Teacher confused by status picker (overridden by recalculation)
- Partial transaction: amount updated but status not (or vice versa)

## 8. Student Summary (Dashboard)

### Purpose
Computed view showing a student's financial health: outstanding balance, active package, upcoming lessons, recent payments, and a status indicator.

### Main Entities
Computed from: `Charge`, `Package`, `LessonParticipant`, `Payment`

### Business Rules
- Only PENDING and PARTIALLY_PAID charges contribute to outstanding balance
- Balance = sum of (charge.amount - sum(allocations)) per charge
- Status: "healthy" (no unpaid), "warning" (unpaid but none overdue), "overdue" (any past dueAt)
- Active package = first ACTIVE package found (only one shown even if multiple exist)
- Upcoming lessons: next 5 future lessons, ordered by scheduledAt ASC
- Recent payments: last 5, ordered by paidAt DESC
- outstandingCurrency: taken from first unpaid charge's currency (single currency assumed)
- 4 parallel queries with no transaction → eventual consistency possible

### Risks if Logic Breaks
- Mixed currencies → incorrect balance display
- Multiple active packages → only first shown
- Stale read from concurrent writes during page load

## 10. Potential Logic Weaknesses Found in Current Implementation

> These are concrete bugs and gaps found by analyzing server actions, Prisma schema cascade behavior, and data flow. Based on actual code, not assumptions.

---

### W-01 — CRITICAL: Package minutes not restored when lesson is deleted

**File**: [src/app/(app)/lessons/actions.ts](src/app/(app)/lessons/actions.ts) — `deleteLesson`

**Root cause**: `prisma.lesson.delete()` triggers cascades:
- Lesson → LessonParticipant (`onDelete: Cascade`)
- LessonParticipant → PackageUsage (`onDelete: Cascade`)

`Package.remainingMinutes` is never updated. Minutes are permanently lost.

**Impact**: Students silently lose pre-paid lesson minutes when a teacher deletes a lesson that had package assignments.

**Fix direction**: Before deleting, query all PackageUsage records for all participants. Restore minutes to each package inside a transaction. Then delete the lesson.

---

### W-02 — CRITICAL: Package minutes not restored when participant is removed

**File**: [src/app/(app)/lessons/actions.ts](src/app/(app)/lessons/actions.ts) — `removeLessonParticipant`

**Root cause**: `prisma.lessonParticipant.delete()` cascades to PackageUsage. The `removePackageFromParticipant` action exists and correctly restores minutes, but `removeLessonParticipant` does NOT call it.

**Impact**: Removing a student from a lesson deletes their package usage but leaves the package's remaining balance unchanged.

**Fix direction**: `removeLessonParticipant` must check for an existing PackageUsage on the participant and restore minutes before (or within the same transaction as) the deletion.

---

### W-03 — CRITICAL: Payment allocation amount not updated when payment amount is edited

**File**: [src/app/(app)/payments/actions.ts](src/app/(app)/payments/actions.ts) — `updatePayment`

**Root cause**: `updatePayment` updates `payment.amount` but does NOT update `PaymentAllocation.amount`. The charge status recalculation in the same transaction reads stale allocation amounts.

**Example**: Payment of CHF 100 allocated to a CHF 100 charge (PAID). Teacher edits payment to CHF 50. Allocation still says 100. Charge recalculates: totalPaid=100 >= 100 → stays PAID. But only 50 CHF was actually paid.

**Impact**: Financial records incorrect. The charge stays PAID even though the payment was reduced.

**Fix direction**: `updatePayment` must also update `PaymentAllocation.amount` for allocations linked to this payment. Requires product decision on strategy (update proportionally? set to payment amount if single allocation?).

---

### W-04 — HIGH: No lesson duration overlap detection

**File**: [src/app/(app)/lessons/actions.ts](src/app/(app)/lessons/actions.ts) — `createLesson`, `updateLesson`

**Root cause**: Conflict check uses exact timestamp equality: `findFirst({ where: { teacherId, scheduledAt } })`.

**Example**: Lesson A at 14:00 for 90 minutes. Lesson B at 14:30. No conflict detected.

**Impact**: Teacher double-booked. Calendar shows overlapping lessons.

**Fix direction**: Implement interval overlap: new lesson's start < existing lesson's end AND new lesson's end > existing lesson's start. Requires computing `scheduledAt + durationMin` as an end time.

### W-05 — HIGH: `addLessonParticipant` does not verify student ownership

**File**: [src/app/(app)/lessons/actions.ts](src/app/(app)/lessons/actions.ts) — `addLessonParticipant`

**Root cause**:

```ts
const student = await prisma.user.findFirst({
  where: { id: userId, role: UserRole.STUDENT },
  // Missing: createdByTeacherId: user.id
```

**Impact**: In multi-teacher scenarios, Teacher A can add Teacher B's student to Teacher A's lesson. Cross-tenant student exposure.

**Fix direction**: Add `createdByTeacherId: user.id` to the student ownership check.

### W-06 — HIGH: No role-based access control after authentication

**File**: [src/lib/auth/require-auth.ts](src/lib/auth/require-auth.ts)

**Root cause**: `requireAuth()` only verifies session existence. Any user role can log in and access teacher functionality.

**Impact**: A STUDENT user with a passwordHash set can log in and access the full teacher UI. Ownership filters (createdByTeacherId) limit some damage but don't prevent all teacher operations.

**Fix direction**: Add `requireTeacherAuth()` that additionally asserts `session.user.role === 'TEACHER'`.

### W-07 — MEDIUM: Silent failure when charge-student mismatch on payment creation

**File**: [src/app/(app)/payments/actions.ts](src/app/(app)/payments/actions.ts) — `createPayment`

**Root cause**:
```ts
if (charge && charge.userId === userId) {
  // create allocation
}
// If mismatch: payment created, no allocation, no error
```

**Impact**: Teacher links a payment to a charge, but if there is a student mismatch, the allocation is silently skipped. Teacher believes the charge is paid but it isn't. No error or warning.

**Fix direction**: Return an explicit error when chargeId is provided but mismatch is detected.

### W-08 — MEDIUM: Zero-amount charges and over-payments accepted without warning

**Files**: `charges/actions.ts`, `payments/actions.ts`

**Root cause**: The decimal regex `/^\d+(\.\d{1,2})?$/` matches "0". No minimum amount validation. No cap on payment vs. remaining charge balance.

**Impact**: A CHF 0 charge with a CHF 0 payment is marked PAID (0 >= 0 is true). Over-payments inflate totalPaid beyond chargeAmount silently.

**Fix direction**: Validate amount > 0 for both charges and payments. Optionally warn when payment exceeds remaining balance on a charge.

### W-09 — MEDIUM: Date format validation does not check calendar validity

**File**: [src/lib/dates.ts](src/lib/dates.ts)

**Root cause**: `isValidDatetimeLocal` checks regex only. `isValidDate` partially validates via `new Date()` but month=13 or Feb 30 can still pass depending on JS engine behavior. `zurichLocalToUtc` silently normalizes invalid dates (Feb 30 → March 2).

**Impact**: Lessons or charges scheduled on impossible dates get silently shifted to wrong dates.

**Fix direction**: After format check, extract and validate month (1–12) and day (within month's actual range).

### W-10 — MEDIUM: `updatePackage` allows reducing total hours below consumed minutes

**File**: [src/app/(app)/packages/actions.ts](src/app/(app)/packages/actions.ts) — `updatePackage`

**Root cause**: No guard that `newTotalMinutes >= (totalMinutes - remainingMinutes)` (consumed minutes).

**Example**: 10h total, 8h consumed (remaining=2h). Edit to 5h. diff=-300. newRemaining=0. Package exhausted. But usage records show 8h consumed from a 5h package.

**Impact**: Usage history becomes historically inconsistent (consumed > total).

**Fix direction**: Validate `newTotalMinutes >= consumed minutes`. Warn teacher if reducing below consumed.

### W-11 — LOW: No rate limiting on login

**File**: [src/lib/auth/actions.ts](src/lib/auth/actions.ts)

**Root cause**: No delay, no attempt counter, no lockout on repeated failed logins.

**Impact**: Login endpoint can be brute-forced.

**Fix direction**: Add exponential backoff, account lockout, or IP-based rate limiting at edge level.

### W-12 — LOW: Sessions accumulate without cleanup

**File**: [src/lib/auth/session.ts](src/lib/auth/session.ts)

**Root cause**: `createSession` always creates a new Session row. Expired sessions are only cleaned up when specifically accessed. No background job or login-time cleanup.

**Impact**: Session table grows unboundedly. Long-running deployments accumulate thousands of expired rows.

**Fix direction**: On successful login, delete expired sessions for the same userId. Or add a scheduled cleanup job.
