## 1. Authentication & Sessions

### Purpose
Controls who can access the application. Uses DB-backed sessions (30-day expiry), bcrypt password hashing, cookie-based session tokens, and a `LoginAttempt` table for rate limiting. Auth is enforced via `requireTeacherAuth()` in every server action and `requireAuth()` in protected layouts.

### Main Entities
`User` (passwordHash, isActive, role), `Session` (expiresAt), `LoginAttempt`

### Business Rules
- Login requires email + password
- Password compared with bcrypt hash stored in User
- Inactive users (`isActive = false`) are rejected even with valid credentials
- Users without a `passwordHash` are rejected (student-only accounts)
- Sessions expire after 30 days
- Expired sessions are deleted on access attempt
- All `(app)` routes protected via layout-level `proxyAuth()` calling `requireAuth()`
- All server actions use `requireTeacherAuth()` — asserts session AND `role === TEACHER`
- Rate limiting: 5 failed login attempts per 15 minutes per email → locked out with message
- On successful login: failed attempts for that email + globally expired attempts pruned
- Alert email sent when rate limit is first hit (5th attempt)

### Critical Validations
- `!user.passwordHash` → rejected
- `!user.isActive` → rejected
- Session cookie presence + DB lookup + expiry check on every authenticated request
- `session.user.role !== 'TEACHER'` in server actions → redirect to `/login`
- `recentFailures >= 5` → "Too many failed sign-in attempts. Please try again in 15 minutes."

### Dependencies
Used by every module. Auth failure redirects to `/login`.

### Risks if Logic Breaks
- Authentication bypass → data exposure
- Role not checked → student user accesses teacher operations
- Rate limit bypass → brute force possible

## 2. Students

### Purpose
Teachers create and manage students. Students are scoped per teacher via `createdByTeacherId`.

### Main Entities
`User` (role=STUDENT, createdByTeacherId, email?, phone?)

### Business Rules
- Student requires firstName + lastName
- At least one contact method: email OR phone
- Email must be globally unique (across all users, all teachers)
- Student is scoped to the creating teacher (`createdByTeacherId = teacher.id`)
- Teacher can only view/edit their own students
- No student deletion action is implemented
- Students can exist without an email (phone only)

### Critical Validations
- `!firstName` → "First name is required."
- `!lastName` → "Last name is required."
- `!email && !phone` → "At least one contact method is required: email or phone."
- `email` format validated by regex `/^[^\s@]+@[^\s@]+\.[^\s@]+$/`
- Email uniqueness checked before insert (application-level pre-check, not just DB constraint)
- On update: self-exclusion `id: { not: id }` applied to uniqueness check

### Dependencies
- Charges, Payments, Packages, Lessons: all reference student via `userId` / `participants`

### Risks if Logic Breaks
- Teacher sees another teacher's students
- Email collisions race condition → DB P2002 unhandled
- Student orphaned records when teacher removed (no cascade guard on `createdByTeacherId`)

## 3. Lessons & Calendar

### Purpose
Teachers schedule lessons with optional student participants. Calendar view shows lessons by day. Scheduling conflict detection prevents overlapping lessons (duration-aware). Three billing modes control what happens financially at lesson creation time.

### Main Entities
`Lesson`, `LessonParticipant`

### Business Rules
- Lesson requires title, scheduledAt (Zurich datetime), durationMin (positive integer)
- priceAmount is optional (decimal ≤ 2 places if provided)
- Conflict detection is duration-aware: a new lesson conflicts if any existing lesson's time range overlaps (not just exact timestamp match)
- Conflict check on update excludes the lesson being edited (self-exclusion via `id: { not: id }`)
- Student participant is optional at lesson creation time
- If no student selected: billingMode is forced to FREE regardless of form value
- Billing mode at creation:
  - `FREE` — lesson created with participant, no charge, no package usage
  - `UNIT` — lesson + participant + Charge (type=LESSON, amount=priceAmount or "0") created atomically
  - `PACKAGE` — lesson + participant + PackageUsage created atomically; package balance decremented; package status updated; packageId required when billingMode=PACKAGE
- Adding a participant post-creation: student must exist, owned by teacher (`createdByTeacherId` checked), status=CONFIRMED; duplicate rejected by P2002
- Removing a participant: restores package minutes IF PackageUsage exists AND no other participant uses the same package for this lesson
- Deleting a lesson: restores package minutes for all participants (iterates usages), then cascade-deletes lesson → participants → usages
- Lesson deletion sets `charge.lessonId` to NULL (SetNull cascade on Charge)

### Critical Validations
- `durationMin`: positive integer, not float, not zero — "Duration must be a positive integer (minutes)."
- `scheduledAt`: must match `YYYY-MM-DDTHH:mm` format (calendar validity NOT checked — JS silently normalizes Feb 30)
- `priceAmount`: decimal regex if provided — "Price must be a valid amount with up to 2 decimals."
- `billingMode === 'PACKAGE' && studentId && !packageId` → "Please select a package."
- Conflict: overlap detection — new lesson end > existing start AND existing end > new start

### Dependencies
- Calendar: lessons displayed by date (Zurich timezone)
- Packages: PackageUsage links LessonParticipant → Package
- Charges: charge.lessonId can reference a lesson
- Dashboard: today's lessons, no-participant alerts, no-package alerts

### Risks if Logic Breaks
- Billing mode creates wrong financial records (double charge, or charge without participant)
- Package minutes consumed at creation not restored if lesson deleted
- Schedule overlap not caught (duration-aware logic broken)

## 4. Packages

### Purpose
Teachers sell hour bundles (packages) to students. Each package creation automatically generates a linked Charge. Package minutes are consumed when assigned to lesson participants. Status transitions reflect consumption and expiry. Packages can have multiple participants.

### Main Entities
`Package`, `PackageUsage`, `Charge` (auto-created on package creation), `PackageParticipant`

### Business Rules
- totalHours must be a positive integer (converted to minutes: × 60)
- remainingMinutes initialized to totalMinutes on creation
- Expiry date must be in the future at creation and at update (if provided)
- Package creation is atomic: Package + PackageParticipant (initial student) + Charge created in one transaction
- Update: new total hours must be >= already consumed hours (guard with error message showing minimum)
- Update adjusts remainingMinutes: `newRemaining = max(remaining + diff, 0)` then capped
- Update status logic:
  - `newRemainingMinutes === 0` → EXHAUSTED
  - `existing.status === EXHAUSTED && newRemaining > 0` → ACTIVE
  - EXPIRED and CANCELED statuses preserved through updates (only EXHAUSTED → ACTIVE reacts)
- No deletion action
- Amount field not editable on update (Charge amount stays at original creation value)

### Critical Validations
- `!userId` → "Student is required."
- `!name` → "Name is required."
- `!totalHours` → "Total hours is required."; not positive integer → "Total hours must be a positive integer."
- `!amount` → "Amount is required."; invalid decimal → "Amount must be a valid number with up to 2 decimals."
- `!currency` → "Currency is required."
- expiresAt in past → "Expiry date must be in the future."
- On update: `newTotalMinutes < consumedMinutes` → "Cannot reduce below already consumed hours (minimum: Xh)."

### Dependencies
- Charges: Package creation auto-creates a Charge (type=PACKAGE)
- PackageUsage: Lessons consume package minutes
- PackageParticipant: defines which students are in the package

### Risks if Logic Breaks
- Atomic transaction failure: Package without Charge or vice versa
- Status stuck in EXHAUSTED after hours increase
- Guard on consumed-hours bypass allows totalMinutes < sum(usages)

## 6. Package Assignment (Lesson → Package)

### Purpose
Assigns a student's package to their participation in a lesson (post-creation assignment via lesson detail page). Consumes minutes atomically with concurrency protection.

### Main Entities
`PackageUsage`, `Package`, `LessonParticipant`

### Business Rules
- Package must be ACTIVE status
- Package must not be expired (`expiresAt < now`, even if status=ACTIVE)
- Package must belong to the same student as the participant (via PackageParticipant membership)
- Package must have remaining minutes > 0
- Minutes consumed = `min(lesson.durationMin, pkg.remainingMinutes)`
- Atomic conditional update: `WHERE remainingMinutes >= minutesConsumed` prevents double-spend
- Only one package per participant (unique constraint on `lessonParticipantId`)
- Remove assignment: restores minutes with `Math.min(remaining + consumed, totalMinutes)` cap
- Remove: EXHAUSTED → ACTIVE after minute restoration; EXPIRED/CANCELED preserved

### Critical Validations
- `pkg.status !== ACTIVE` → DomainError "Package is not active."
- `pkg.expiresAt < now` → DomainError "Package has expired."
- No PackageParticipant membership → DomainError "Package does not belong to this student."
- `pkg.remainingMinutes <= 0` → DomainError "Package has no remaining minutes."
- `updated.count === 0` → DomainError "Package no longer has sufficient minutes. It may have been modified concurrently."

## 9. Student Summary (Dashboard)

### Purpose
Computed view showing a student's financial health: outstanding balance, active package, upcoming lessons, recent payments, and a status indicator.

### Main Entities
Computed from: `Charge`, `Package`, `LessonParticipant`, `Payment`

### Business Rules
- Only PENDING and PARTIALLY_PAID charges contribute to outstanding balance
- Balance = sum of (charge.amount - sum(allocations)) per charge
- Status: "healthy" (no unpaid), "warning" (unpaid but none overdue), "overdue" (any past dueAt)
- Active package = first ACTIVE package found (ordered by createdAt DESC)
- Upcoming lessons: next 5 future lessons (scheduledAt >= now), ordered by scheduledAt ASC
- Recent payments: last 5, ordered by paidAt DESC
- outstandingCurrency: taken from first unpaid charge's currency (single currency assumed)
- 4 parallel queries with no transaction → eventual consistency possible
- Packages shown: ALL packages for this student (all statuses), ordered by createdAt DESC

### Risks if Logic Breaks
- Mixed currencies → incorrect balance display
- Multiple active packages → only first shown in summary card
- Stale read from concurrent writes during page load

## 10. Dashboard Home & Alerts

### Purpose
The teacher dashboard shows today's lessons, pending charges summary, and contextual alerts about packages and upcoming lessons needing attention.

### Main Entities
Computed from: `Lesson`, `Package`, `Charge`, `PaymentAllocation`, `PackageUsage`, `PackageParticipant`

### Business Rules
- Today's lessons: filtered by Zurich-day boundaries (not UTC midnight)
- Pending charges: top 5 by createdAt DESC, PENDING or PARTIALLY_PAID only
- Total owed: all pending/partially-paid charges (not just top 5), single currency assumed
- Alert types:
  - `package_low`: ACTIVE packages with remainingMinutes < 120 (< 2h)
  - `package_expiring`: ACTIVE packages with `expiresAt` within next 7 days (gte now, lte now+7d)
  - `lesson_no_participant`: future lessons (up to 20) with no participants
  - `lesson_no_package`: future lessons with participants who have no PackageUsage assigned
- `package_low` and `package_expiring` can both fire for the same package simultaneously
- `lesson_no_package` fires per participant (one lesson can generate multiple alerts)
- Alerts are not persisted; computed fresh on each dashboard load

### Risks if Logic Breaks
- Zurich day boundaries computed incorrectly → today's lessons missing or showing wrong day
- `package_low` threshold hardcoded at 120min — not configurable
- Multi-participant packages: studentName for alerts = all participant names joined by ", "

## 12. Potential Logic Weaknesses Found in Current Implementation

> These are concrete bugs and gaps found by analyzing server actions, Prisma schema cascade behavior, and data flow. Based on actual code, not assumptions.
> Items marked [FIXED] have been resolved in the codebase as of the last analysis.

---

### W-01 — MEDIUM: Multi-currency balance in Student Summary and Dashboard

**Files**: `src/features/students/queries/getStudentDetail.ts:195-200`, `src/features/dashboard/queries.ts:131-137`

**Root cause**: Outstanding balance is computed by summing all unpaid charge amounts numerically. Currency label is taken from the first charge. If a student has charges in multiple currencies (e.g., CHF and EUR), the displayed total is numerically incorrect.

**Impact**: Teacher sees "CHF 180" when the actual breakdown is CHF 100 + EUR 80.

**Fix direction**: Either enforce single-currency per teacher/student, or compute balance per currency and display multiple totals.

---

### W-02 — HIGH: Over-payment on charges accepted silently

**Files**: `src/app/(app)/charges/actions.ts`, `src/app/(app)/payments/actions.ts`

**Root cause**: No cap on payment allocation vs. remaining charge balance. `totalPaid >= amount` is the PAID condition, so any amount over the charge is silently accepted.

**Impact**: Teacher may over-allocate payments, inflating the "paid" amount on a charge with no warning.

**Fix direction**: Validate that payment amount does not exceed remaining balance when linking to a charge. Optionally warn on over-payment.

---

### ~~W-03 — MEDIUM: Invalid date components pass format check~~ [FIXED]

**File**: `src/lib/dates.ts`

**Root cause**: `isValidDatetimeLocal` checks regex format only. `zurichLocalToUtc` silently normalizes invalid dates (Feb 30 → March 2).

**Impact**: Lessons or charges scheduled on impossible dates get silently shifted to wrong dates.

**Fix direction**: After format check, validate month (1–12) and day (within month's actual range).

**Resolution**: `isValidDatetimeLocal` now validates calendar dates via Date round-trip check (`d.getUTCDate() === dy`). Feb 30 normalises to March 2, which fails the check — validation error returned.

---

### W-04 — LOW: Sessions accumulate without full cleanup

**File**: `src/lib/auth/session.ts`

**Root cause**: `createSession` always creates a new Session row. On successful login, only expired rows for the same email + globally expired rows are pruned. Non-expired old sessions for the same userId remain (concurrent device support by design, but old active sessions from the same device are not purged).

**Impact**: Session table grows over time. Long-running instances accumulate rows.

**Fix direction**: Optionally purge old sessions for the same userId on new login (single-device mode). Or add a scheduled cleanup job for rows older than N days.

---

### W-05 — MEDIUM: Lesson detail page has no teacher ownership guard

**File**: `src/app/(app)/lessons/[id]/page.tsx:17`

**Root cause**: `prisma.lesson.findUnique({ where: { id } })` — no `teacherId` filter. Any authenticated teacher can view any lesson's detail page.

**Impact**: Cross-tenant read exposure. Teacher A can view Teacher B's lesson with participant names, booking statuses, and package assignments.

**Fix direction**: Add `where: { id, teacherId: session.user.id }` and redirect to `notFound()` if not found.

---

### W-06 — MEDIUM: billingMode=PACKAGE at lesson creation does not throw on invalid package

**File**: `src/app/(app)/lessons/actions.ts:244-262`

**Root cause**: The inline PACKAGE billing logic inside `createLesson` silently skips package deduction if `pkg` is null, not found, or `remainingMinutes <= 0`. No error is thrown or returned.

**Impact**: Lesson and participant are created, teacher selects PACKAGE mode and a packageId, but if the package was exhausted or belongs to wrong student, no usage is recorded and no error is shown. Teacher believes the package was consumed.

**Fix direction**: Validate package availability before the transaction (like `assignPackageToParticipant` does), or throw `DomainError` inside the transaction branch.
